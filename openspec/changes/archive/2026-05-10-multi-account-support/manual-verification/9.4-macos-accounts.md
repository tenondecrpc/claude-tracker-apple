# 9.4 Manual Verification - macOS Multi-Account Add / Switch / Sign Out

This document is the manual verification procedure for task 9.4 of the
`multi-account-support` change. It covers the macOS menu bar app only.
A human tester MUST execute each step in order and mark the acceptance
checklist at the end.

Objective: confirm that adding two accounts, switching the active account,
and signing out of the currently active account produces the expected
per-account Keychain slots and iCloud directories, and that registry +
active-account state stays consistent across the flow.

Bugfix / pure-code changes are not in scope. This task intentionally does
not assert UI pixel layout; it asserts behavior on three persistence
surfaces: the macOS Keychain, the iCloud ubiquity container, and the
visible account state in the menu bar popover + Preferences.

## Preconditions

Environment:

- macOS 15 or later on a test machine that is NOT a production user's
  daily driver. The legacy credential cleanup in
  `Tempo macOS/LegacyCredentialsCleanup.swift` will delete any old
  single-slot credentials on first launch; this is intentional per
  `design.md` ("No migration, no fallback").
- The `Tempo macOS` scheme built fresh from the current branch in Xcode,
  launched with its entitlements intact. Do NOT install a TestFlight build
  for this verification.
- iCloud Drive signed in under a user that has the
  `iCloud.com.tenondev.tempo.claude` container available. From Finder,
  `~/Library/Mobile Documents/` MUST contain a directory whose name
  encodes that container, typically
  `iCloud~com~tenondev~tempo~claude/Documents/Tempo/`.
- Access to two real Anthropic accounts that can complete the Claude Code
  OAuth flow end to end. Referred to below as:
  - Account A: first account. Expected canonical accountId:
    `accountA@example.com` (lowercased email).
  - Account B: second account. Expected canonical accountId:
    `accountB@example.com` (lowercased email).
  - Substitute the real email addresses in place of these placeholders.
- Terminal access to run `security` and `ls` against the local Keychain
  and iCloud path.
- An app password or other unlocking flow ready for the login keychain
  (the OS may prompt when the app calls `SecItem*`).

Starting state:

- Quit any previous `Tempo` process (menu bar icon gone).
- Manually clear any stale iCloud state from prior dev builds (optional
  but recommended):
  - In Finder, navigate to
    `~/Library/Mobile Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/`
    and delete the `accounts/` directory if present.
  - Delete any leftover `usage.json`, `usage-history.json`, or
    `latest.json` siblings in the same `Tempo/` folder; the multi-account
    build does not read them but their presence can confuse verification.
- In Keychain Access, search for `com.tenondev.tempo.claude.oauth` and
  delete every item that matches. There should be at most one legacy
  `credentials` slot plus any prior per-account slots from earlier runs.
  (Alternatively, run the Terminal command in the cleanup helper below
  before launching the app.)

Optional cleanup helper (Terminal, before launch):

```sh
security delete-generic-password -s com.tenondev.tempo.claude.oauth -a credentials 2>/dev/null || true
security delete-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__ 2>/dev/null || true
```

The app's one-shot `LegacyCredentialsCleanup.sweep()` will also remove
the legacy `credentials` slot and any
`~/.config/tempo-for-claude/credentials.json` file on first launch, but
manually clearing the Keychain first keeps the verification output
uncluttered.

## Definitions

- Keychain service used for all OAuth slots:
  `com.tenondev.tempo.claude.oauth`
- `kSecAttrAccount` values:
  - Canonical accountId (lowercased email, NFC, trimmed) for a real
    account. See `Shared/AccountIdentifier.swift`.
  - Reserved string `__registry__` for the non-secret account list.
    Owned by `AccountRegistry` and NEVER a credential.
- iCloud root path on the test Mac:
  `~/Library/Mobile Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/`
  Referred to below as `<iCloudTempo>`.
- Per-account directory: `<iCloudTempo>/accounts/<percentEncodedAccountId>/`.
  Characters outside `[a-z0-9._@-]` are percent-encoded by
  `AccountIdentifier.percentEncodedDirectoryName(for:)`. The `@` in a
  typical email is preserved as `@` in the directory name.
- Per-account files:
  - `account.json` - non-secret metadata (written by
    `AccountRegistryICloudMirror`)
  - `usage.json` - current `UsageState` (written by the polling worker)
  - `usage-history.json` - per-account history
  - `latest.json` - latest relayed Claude Code session event
- Discovery index: `<iCloudTempo>/accounts/index.json`

## Procedure

Execute each step. After each step, record the on-disk output using the
inspection commands in the "Reference commands" section below.

### Step 1 - Fresh launch with no accounts

1. Launch the `Tempo macOS` build from Xcode (`Cmd+R`) or from the built
   `.app` bundle. The menu bar icon should appear.
2. Open the menu bar popover. The Account row should read
   `Not signed in`. The content area should show the "Not Signed In"
   state (no usage ring, sign-in prompt).
3. Verify Keychain has NO `com.tenondev.tempo.claude.oauth` items yet
   (the `__registry__` slot is created lazily on first `registry.add`):

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth 2>&1 | head -5
   ```

   Expected: `security: SecKeychainSearchCopyNext: The specified item
   could not be found in the keychain.`
4. Verify iCloud `<iCloudTempo>/accounts/` does NOT exist yet:

   ```sh
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts 2>&1
   ```

   Expected: `No such file or directory`.

### Step 2 - Add account A

1. In the popover, click the Account row to open the menu.
2. Click `Add account...`. The menu bar popover closes and the Welcome
   window opens.
3. Complete the OAuth flow for account A (`accountA@example.com`).
4. When the Welcome window reports success, close it.
5. Open the popover again. Confirm:
   - Account row now displays the account A label (displayName falls
     back to the email when the OAuth profile does not expose a display
     name).
   - Usage ring starts fetching (spinner) or already shows data.
   - `Logout` row in the action items shows account A's email as the
     subtitle.
6. Open the `Stats` detail window. Confirm the header identifies
   account A.

Inspect persistence:

7. Keychain: exactly two items under service
   `com.tenondev.tempo.claude.oauth` should now exist. One has
   `kSecAttrAccount = accountA@example.com`, and one has
   `kSecAttrAccount = __registry__`.

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountA@example.com 2>&1 | head -15
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__ 2>&1 | head -15
   ```

   Each call MUST succeed (no "item could not be found" error). Do NOT
   dump the password data; the goal is only to confirm the slot exists.

8. iCloud: the per-account directory for A should exist.

   ```sh
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/accountA@example.com/
   ```

   Expected listing for the per-account directory (files may appear
   progressively as iCloud finishes uploading and the first poll lands):
   - `account.json` (should appear within a few seconds)
   - `usage.json` (appears after the first successful poll, typically
     within 15 to 60 seconds)
   - `usage-history.json` (appears after the first poll)
   - `latest.json` may or may not appear depending on whether any Claude
     Code session has been ingested during the run.

9. iCloud index file: `<iCloudTempo>/accounts/index.json` MUST exist and
   contain `accountA@example.com` in `accountIds`:

   ```sh
   cat ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/index.json
   ```

   Expected shape:

   ```json
   {
     "accountIds" : [ "accountA@example.com" ],
     "updatedAt" : "2025-..."
   }
   ```

### Step 3 - Add account B

1. Open the popover. Click the Account row to open the menu.
2. Click `Add account...`. Complete OAuth for account B
   (`accountB@example.com`).
3. When Welcome reports success, reopen the popover. Confirm:
   - Account row displays account B (B becomes the active account after
     sign-in because `submitOAuthCode` calls
     `registry.setActive(accountId:)` for the new account).
   - The account menu now lists A with a `Set as active: accountA@...`
     entry and no entry for B (the active account is not listed as a
     "Set as active" target).
4. Open Preferences via the popover's `Preferences` row, then the
   `Accounts` card. Confirm both accounts appear as rows, account B is
   marked `Active`, and each row shows:
   - Display name (email as fallback)
   - Email
   - `Added ...` timestamp
   - `Last poll: ...` (relative time, or `never` if the worker has not
     yet run)
   - `Last session: -` (placeholder per the current code; not part of
     the acceptance criteria)

Inspect persistence:

5. Keychain: three items total under service
   `com.tenondev.tempo.claude.oauth`:
   - `kSecAttrAccount = accountA@example.com`
   - `kSecAttrAccount = accountB@example.com`
   - `kSecAttrAccount = __registry__`

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountA@example.com 2>&1 | grep '"acct"'
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountB@example.com 2>&1 | grep '"acct"'
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__ 2>&1 | grep '"acct"'
   ```

6. iCloud: both per-account directories exist.

   ```sh
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/
   ```

   Expected: two subdirectories (`accountA@example.com/` and
   `accountB@example.com/`) plus `index.json`.

7. Index now lists both ids, in registry order (A first, B second):

   ```json
   {
     "accountIds" : [ "accountA@example.com", "accountB@example.com" ],
     "updatedAt" : "2025-..."
   }
   ```

### Step 4 - Switch active account (B to A, then A to B)

1. In the popover's Account menu, click `Set as active: account A`.
2. Confirm the Account row label updates to account A immediately.
3. Confirm the usage ring either refreshes to A's latest snapshot
   (already cached) or begins polling A.
4. Open the `Stats` detail window. Confirm the header now identifies
   account A, and the usage, history, and session content rebind to A.
5. In the popover's Account menu again, click `Set as active: account B`.
6. Confirm the Account row label flips back to account B and the
   detail window reloads for B.

Persistence after the switch:

7. No Keychain items are added or removed during a switch. Re-run the
   three `security find-generic-password` commands from Step 3.5 and
   confirm the same three slots are still present.
8. No per-account iCloud directories are added or removed during a
   switch. Re-run the `ls` from Step 3.6 and confirm both accounts are
   still present.
9. `index.json` content is unchanged by a switch (it mirrors the
   registry membership, not the active selection). The
   `updatedAt` timestamp may or may not advance on every click depending
   on whether the coordinator re-mirrors after a `setActive`; both
   outcomes are acceptable for this verification.
10. The active-account selection is persisted in macOS `UserDefaults`
    under key `tempo.activeAccountId` (not in iCloud). Spot-check:

    ```sh
    defaults read com.tenondev.tempo.claude tempo.activeAccountId
    ```

    Expected: the current active accountId (the app's real bundle id
    may differ; use the value from
    `Bundle.main.bundleIdentifier` printed from the app or from the
    actual `plist` file under `~/Library/Preferences/`). A missing key
    after a switch is a failure; the value not being iCloud-synced is
    expected and correct.

### Step 5 - Sign out of the active account (account B)

With account B currently active (per Step 4):

1. Open `Preferences` and scroll to the `Accounts` card.
2. On account B's row, click `Sign out`. (Alternative: use the popover
   `Logout` action in the action items, which currently signs out the
   active account only. Either path exercises the same
   `MacOSAPIClient.signOut(for:)` code.)
3. Observe that:
   - Account B disappears from the Preferences Accounts list.
   - Account A remains listed and is now marked `Active`.
   - The menu bar popover's Account row updates to account A.
   - The Welcome window does NOT open automatically because at least
     one account remains authenticated
     (`requiresExplicitSignIn` stays `false` while `accounts` is
     non-empty).

Persistence after sign-out:

4. Keychain: account B's slot is gone. Account A and `__registry__` are
   still present.

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountB@example.com 2>&1 | head -2
   ```

   Expected: `security: SecKeychainSearchCopyNext: The specified item
   could not be found in the keychain.`

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountA@example.com 2>&1 | grep '"acct"'
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__ 2>&1 | grep '"acct"'
   ```

   Both MUST still succeed.

5. iCloud: account B's per-account directory is removed.

   ```sh
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/
   ```

   Expected: only `accountA@example.com/` remains alongside `index.json`.
   Note: iCloud sync has a short delay; allow up to ~30 seconds for the
   removed directory to disappear from the Finder-visible path. The
   `AccountRemovalService` calls `FileManager.removeItem(at:)`
   synchronously, so the file SHOULD disappear from the local mirror
   immediately. If it persists past a minute, this is a failure.

6. Index file: `accountIds` no longer contains account B.

   ```sh
   cat ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/index.json
   ```

   Expected:

   ```json
   {
     "accountIds" : [ "accountA@example.com" ],
     "updatedAt" : "2025-..."
   }
   ```

   `updatedAt` MUST have advanced relative to the pre-sign-out value.

7. Active-account pointer: `tempo.activeAccountId` is either
   `accountA@example.com` (promoted from the first remaining account)
   or unset if for some reason no accounts remain. Because A was not
   signed out in this step, the expected value is
   `accountA@example.com`.

### Step 6 - Sign out of the last remaining account

With account A now the only account and active:

1. In Preferences, on account A's row, click `Sign out`.
2. Observe that:
   - The Accounts card reverts to `No accounts signed in`.
   - The menu bar popover Account row reads `Not signed in`.
   - The Welcome window opens automatically (`requiresExplicitSignIn`
     flips `true` when `accounts` becomes empty).

Persistence after sign-out of the last account:

3. Keychain: no credential slots remain. The `__registry__` slot still
   exists because the registry is now persisted as an empty list, not
   deleted.

   ```sh
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountA@example.com 2>&1 | head -2
   security find-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__ 2>&1 | grep '"acct"'
   ```

   Expected: the first call reports "item could not be found", the
   second succeeds and shows the `__registry__` slot.

4. iCloud: the per-account directory for A is removed.

   ```sh
   ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/
   ```

   Expected: `accounts/` contains only `index.json` (no subdirectories)
   after iCloud settles.

5. Index file: `accountIds` is empty.

   ```json
   {
     "accountIds" : [],
     "updatedAt" : "2025-..."
   }
   ```

6. `tempo.activeAccountId` is cleared (the key is removed from
   `UserDefaults`).

## Reference commands

Quick-copy commands for the human tester. Replace placeholder addresses
with the real ones.

Keychain inspection:

```sh
# List all OAuth service items the app owns (dumps metadata, not the
# secret payload).
security dump-keychain -a login.keychain-db 2>/dev/null \
  | awk '/"svce"/{svce=$0} /"acct"/{print svce; print $0; print "---"}' \
  | grep -B1 'com.tenondev.tempo.claude.oauth'

# Confirm a specific accountId slot exists without printing the password.
security find-generic-password -s com.tenondev.tempo.claude.oauth -a accountA@example.com

# Confirm the registry slot exists.
security find-generic-password -s com.tenondev.tempo.claude.oauth -a __registry__
```

iCloud inspection:

```sh
# List the accounts tree.
ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/

# List one account's directory.
ls -la ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/accountA@example.com/

# Dump the index file.
cat ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/index.json

# Dump a per-account metadata file.
cat ~/Library/Mobile\ Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/accountA@example.com/account.json
```

Active-account pointer:

```sh
# Replace com.tenondev.tempo with the actual bundle id if different
# (e.g. the TestFlight id). `defaults find` can help locate it.
defaults read com.tenondev.tempo tempo.activeAccountId
```

## Expected layouts

### Keychain (after Step 3, two accounts added)

Under service `com.tenondev.tempo.claude.oauth`:

| `kSecAttrAccount`        | Owner               | Contents                                                |
|--------------------------|---------------------|---------------------------------------------------------|
| `accountA@example.com`   | `CredentialStore`   | JSON-encoded `StoredCredentials` (OAuth tokens for A)   |
| `accountB@example.com`   | `CredentialStore`   | JSON-encoded `StoredCredentials` (OAuth tokens for B)   |
| `__registry__`           | `AccountRegistry`   | JSON-encoded `[Account]` (non-secret registry payload)  |

There MUST NOT be a legacy `credentials` slot after first launch; if one
appears it is a bug in `LegacyCredentialsCleanup`.

### iCloud (after Step 3, two accounts added)

```
~/Library/Mobile Documents/iCloud~com~tenondev~tempo~claude/
  Documents/
    Tempo/
      accounts/
        index.json                         <- AccountsIndexFile, two ids
        accountA@example.com/
          account.json                     <- non-secret metadata for A
          usage.json                       <- current UsageState for A (after first poll)
          usage-history.json               <- per-account history for A
          latest.json                      <- optional; present once a session is ingested
        accountB@example.com/
          account.json
          usage.json
          usage-history.json
          latest.json                      <- optional
      alert-preferences.json               <- global, unchanged by this flow
      appearance-mode.json                 <- global, unchanged by this flow
```

`accounts/` MUST NOT contain any subdirectory that does not map back to
an entry in `index.json.accountIds`.

Files that MUST NOT be present under `Tempo/` (legacy layout):
- `Tempo/usage.json`
- `Tempo/usage-history.json`
- `Tempo/latest.json`

If any of those appear during the run, that is a regression against task
3.7 ("Remove any code paths that read or write the legacy flat ... files").
They may be leftover from older dev builds; in that case delete them
manually and re-run the verification.

### iCloud (after Step 5, active account signed out)

```
Tempo/
  accounts/
    index.json                             <- accountIds: ["accountA@example.com"]
    accountA@example.com/
      account.json
      usage.json
      usage-history.json
      latest.json                          <- optional
  alert-preferences.json
  appearance-mode.json
```

### iCloud (after Step 6, last account signed out)

```
Tempo/
  accounts/
    index.json                             <- accountIds: []
  alert-preferences.json
  appearance-mode.json
```

## Acceptance checklist

Check every box before marking task 9.4 complete. If any box cannot be
checked, record the observation under `Notes` and leave the task in the
in-progress state.

Step 1 (fresh launch):

- [ ] Popover shows `Not signed in` Account row on first launch.
- [ ] No `com.tenondev.tempo.claude.oauth` Keychain items exist.
- [ ] `<iCloudTempo>/accounts/` directory does not exist.

Step 2 (add account A):

- [ ] Welcome window completes OAuth for account A without error.
- [ ] Popover Account row displays account A after sign-in.
- [ ] Keychain slot `kSecAttrAccount = accountA@example.com` exists.
- [ ] Keychain slot `kSecAttrAccount = __registry__` exists.
- [ ] `<iCloudTempo>/accounts/accountA@example.com/account.json` exists.
- [ ] `<iCloudTempo>/accounts/index.json` lists exactly
      `accountA@example.com`.
- [ ] Within roughly 60 seconds of sign-in, `usage.json` and
      `usage-history.json` appear in A's directory.

Step 3 (add account B):

- [ ] Adding B promotes B to active (Account row label is account B).
- [ ] Account A is still present in the Preferences Accounts list.
- [ ] Keychain slots A, B, and `__registry__` all exist.
- [ ] `<iCloudTempo>/accounts/` contains both
      `accountA@example.com/` and `accountB@example.com/` subdirectories.
- [ ] `index.json.accountIds` contains both ids.

Step 4 (switch A, then B):

- [ ] `Set as active` switches the popover's Account row label.
- [ ] The Stats detail window rebinds to the newly active account.
- [ ] Keychain slot set is unchanged by a switch.
- [ ] `accounts/` directory set is unchanged by a switch.
- [ ] `tempo.activeAccountId` in `UserDefaults` tracks the active
      accountId after each switch.

Step 5 (sign out of active account B):

- [ ] Account B disappears from the Preferences Accounts list.
- [ ] Account A is promoted to active automatically.
- [ ] Keychain slot `accountB@example.com` is gone.
- [ ] Keychain slots `accountA@example.com` and `__registry__` are
      still present.
- [ ] `<iCloudTempo>/accounts/accountB@example.com/` is removed (within
      about 30 seconds of iCloud sync; local write is synchronous).
- [ ] `index.json.accountIds` no longer contains account B.
- [ ] Welcome window does NOT open automatically because account A
      remains authenticated.

Step 6 (sign out of last remaining account A):

- [ ] Popover Account row reverts to `Not signed in`.
- [ ] Welcome window opens automatically.
- [ ] No per-account Keychain credential slots remain under
      `com.tenondev.tempo.claude.oauth`.
- [ ] `__registry__` Keychain slot still exists (with an empty list
      payload).
- [ ] `<iCloudTempo>/accounts/` contains only `index.json` (no
      subdirectories).
- [ ] `index.json.accountIds` is the empty array `[]`.
- [ ] `tempo.activeAccountId` is removed from `UserDefaults`.

Global invariants (check once at the end):

- [ ] At no point during the run did a file named `usage.json`,
      `usage-history.json`, or `latest.json` appear directly under
      `<iCloudTempo>/` (legacy flat layout).
- [ ] No Keychain items appeared with `kSecAttrAccount = credentials`
      after the first launch.
- [ ] No OAuth tokens or refresh tokens are present anywhere in
      `<iCloudTempo>/`. `account.json` MUST contain only `accountId`,
      `email`, `displayName`, `createdAt`.
