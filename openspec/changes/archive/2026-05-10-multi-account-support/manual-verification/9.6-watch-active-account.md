# 9.6 Manual Verification - Watch Follows iPhone Active Account, NoActiveAccount Clearing, Completion Alert Suppression

This document is the manual verification procedure for task 9.6 of the
`multi-account-support` change. It covers the watchOS app
(`Tempo Watch` scheme) running on a real Apple Watch paired to the
physical iPhone from 9.5, plus the iOS relay
(`Tempo/WatchRelayManager.swift`) and macOS session ingestion
(`Tempo macOS`). A human tester MUST execute each step in order and
mark the acceptance checklist at the end.

Objective: confirm that the watch tracks whichever account the iPhone
currently has active (`UsageState` header label and on-wrist ring
rebind on switch), that the watch clears its per-account state and
renders the "No accounts available" placeholder when the iPhone
relays `NoActiveAccount`, that a completion event for an account that
is NOT the iPhone's active account does NOT fire a haptic or a local
notification on the watch while the active-account session DOES fire
one, that the special `unassigned` (CLI-only) bucket is exempt from
the gate, and that a pending completion sheet queued for account A on
the watch is dropped when the iPhone switches to account B before
the sheet fires.

This task is watch-side behavior driven by iPhone relay output. It
consumes the per-account iCloud tree produced by macOS in 9.4 and the
iOS relay wired up in 9.5. Do NOT attempt to exercise this on the
watch Simulator or on a paired iPhone Simulator: `WCSession` pairing
is incomplete on the simulator, `iCloudUsageReader` early-exits on
`targetEnvironment(simulator)`, and the watch's
`UNUserNotificationCenter` authorization flow needs a real wrist to
confirm haptic delivery. Use a physical iPhone paired to a physical
Apple Watch.

## Preconditions

Environment:

- Physical Apple Watch paired to the physical iPhone from 9.5. The
  watch MUST be on watchOS 26 or later (matching the `Tempo Watch`
  scheme's deployment target) and signed into the same iCloud account
  as its iPhone and the macOS test machine from 9.4.
- The `Tempo Watch` scheme built fresh from the current branch in
  Xcode and installed on the watch via `Cmd+R` with the Apple Watch
  selected as the run destination. Do NOT install a TestFlight build
  for this verification. A TestFlight install is acceptable only if it
  corresponds to the exact branch under test.
- The `Tempo` iOS app from 9.5 is installed on the paired iPhone and
  the iPhone is currently showing account A active per the end state
  of 9.5 (chip labeled `accountA@example.com`). If 9.5 left the iPhone
  on account B, switch back to A before starting this procedure.
- The `Tempo macOS` app from 9.4 is running on the paired Mac with
  both accounts signed in (A and B). macOS is what drives fresh
  `latest.json` writes when Claude Code sessions complete; without it
  you cannot exercise the completion alert paths in Steps 4 and 5.
- Watch complication: optional. If the user has the circular
  `utilization5h` complication installed on their current watch face
  from prior builds, it is a useful side-signal but is not a gate for
  any step in this procedure. The complication is driven by
  `TempoWatchShared.complicationUtilization5hKey` and reloads via
  `WidgetCenter.shared.reloadAllTimelines()` on every `UsageState`
  delivery (see `WatchSessionReceiver.applyUsageState`).
- Watch notifications permission: on first launch after install, the
  watch presents an `UNUserNotificationCenter.requestAuthorization`
  prompt (driven by
  `WatchAlertManager.syncAuthorization(enabledInPreferences:)` when
  `watchAlertsEnabled` is true). Grant permission; the rest of this
  procedure assumes authorization is `authorized`. Verify via
  Watch > Settings > Notifications > Tempo before starting Step 4.
- Alert preferences from 9.5 / prior builds: in the iOS `Tempo` app,
  Settings tab > Alerts, ensure `Watch alerts` is ON. This flips
  `SessionAlertPreferences.watchAlertsEnabled = true` which is what
  the watch reads via `watchAlertsEnabled` on both `UsageState` and
  `SessionInfo` payloads. If watch alerts are OFF, Steps 4 and 5
  both produce no haptic regardless of active-account gating, which
  would make the active-account test vacuously "pass".
- Ability to trigger real Claude Code session completions on the Mac
  under each account. The simplest path is:
  - Use the Claude Code CLI with account A signed in (CLI
    `oauthAccount.emailAddress == accountA@example.com`); run a short
    prompt and let it finish. This causes
    `ClaudeLocalDBReader` + `SessionEventWriter` in `Tempo macOS` to
    write `Tempo/accounts/accountA@example.com/latest.json`, which
    iOS `iCloudUsageReader` picks up and hands to
    `WatchRelayManager.sendSession`.
  - Then switch the CLI to account B (`claude logout` then
    `claude login` with the other Anthropic account, or whatever
    account-switch flow the user normally uses with the CLI), run
    another short prompt, and let it finish.
  - An `unassigned` session is produced when the CLI's
    `~/.claude.json` `oauthAccount.emailAddress` does not match any
    account in the macOS `AccountRegistry` (for example, if the CLI
    is signed in as an account that is NOT currently added in the
    macOS Tempo app, or if the OAuth profile lacks an email and the
    synthetic `cli-local-...` id applies). In that case the macOS
    writer drops the session into
    `Tempo/accounts/unassigned/latest.json` and
    `SessionInfo.accountId` equals
    `AccountIdentifier.unassignedAccountId` (`"unassigned"`).
- Access to the iPhone console via `Console.app` (Mac, with the
  iPhone and the paired Apple Watch selected in the device list).
  This is where `AlertTrace` DevLog lines from `WatchRelayManager`,
  `WatchSessionReceiver`, and `TokenStore` are visible. The key
  messages to watch for are called out inline in each step.

Starting state:

- Watch on-wrist, unlocked, and running the `Tempo Watch` app on the
  Dashboard tab (the first page of the vertical `TabView` in
  `RootView`).
- The dashboard shows account A's ring, the header label in the
  tappable account chip reads a short form of
  `accountA@example.com` (typically `accountA`, the email local-part
  per `ContentView.shortAccountLabel()`), and the "No accounts
  available" placeholder is NOT visible.
- No `pendingCompletion` sheet is currently presented. If one is
  visible from a previous run, tap through it to dismiss and confirm
  the dashboard returns.

## Definitions

- `WCSession` channels used in this flow:
  - `updateApplicationContext(_:)`: replace-current-state channel.
    Carries `UsageState`, `AppearanceMode`, and the
    `NoActiveAccount` marker. Per design.md ("iPhone's
    `activeAccountId` is the single source of truth for the watch")
    this is what flips the watch header label and the "No accounts
    available" state.
  - `transferUserInfo(_:)`: durable queued delivery channel. Carries
    `SessionInfo` completion payloads. Survives a watch reboot or
    brief disconnect.
- Active-account header label: the small capsule above the ring in
  `Tempo Watch/ContentView.swift` that renders
  `ContentView.shortAccountLabel()` (email local-part when the label
  looks like an email; initials for multi-word display names;
  truncated prefix otherwise). Tapping it shows
  `accountDetailSheet` with the full label.
- `TokenStore.hasNoActiveAccount` (`Tempo Watch/TokenStore.swift`):
  `true` when the iPhone has relayed a `NoActiveAccount` payload and
  the watch has not since received a `UsageState`. The Dashboard
  branches on this: when `true` it renders `noAccountsState`
  ("No accounts available" / "Check the Mac app to add one"); when
  `false` it renders the usage ring.
- `TokenStore.activeAccountId` and `TokenStore.accountLabel`:
  `@Observable` properties written by
  `TokenStore.apply(_:)` and
  `TokenStore.applyActiveAccount(id:label:)`. Used by the header and
  by the completion sheet's "for <label>" footer in
  `CompletionView`.
- `AccountIdentifier.unassignedAccountId`: the sentinel string
  `"unassigned"` from `Shared/AccountIdentifier.swift`. Sessions with
  this `accountId` are CLI-only and exempted from the watch
  active-account gate in both `WatchRelayManager.sendSession` (iOS
  side) and `WatchSessionReceiver.applySessionInfo` (watch side).
  `UsageState` decoding does NOT fall back to this value.
- Two-layer completion alert gate:
  - Outer gate in `WatchRelayManager.sendSession`: when
    `sessionInfo.accountId != activeAccountId` the iOS relay drops
    the `transferUserInfo` call outright. The session is neither
    delivered to the watch nor recorded against
    `lastRelayedSessionID`, so if the user later sets that account
    active, a subsequent `sendSession` call delivers normally. This
    is the primary mechanism preventing a non-active completion
    from ever reaching the watch.
  - Inner gate in `WatchSessionReceiver.applySessionInfo`: defense
    in depth. Even if a `SessionInfo` with a mismatched `accountId`
    arrives (older queued `transferUserInfo` from before the user
    switched accounts, for example), the receiver refuses to mutate
    `sessions` history, refuses to raise `pendingCompletion`, and
    refuses to call `WatchAlertManager.notifySessionCompletion`.
    Only ambient fields (`appearanceMode`,
    `watchAlertsEnabledInPreferences`) are applied so unrelated
    settings do not get starved.
  - Exemption: `session.accountId ==
    AccountIdentifier.unassignedAccountId` bypasses both gates per
    design.md. Unassigned CLI sessions always deliver to whatever
    account the watch is currently following.
- Stale `pendingCompletion` clearing:
  `TokenStore.clearPendingCompletionIfStale(newAccountId:)` is called
  from `apply(_:)` and `applyActiveAccount(id:label:)`. When a new
  non-empty `accountId` arrives and the watch has a `pendingCompletion`
  whose `accountId` does not match, the sheet is discarded before the
  new state commits. The corresponding `AlertTrace` line is
  `"Clearing stale pendingCompletion: pending accountId=<A> != new=<B>"`.

## Procedure

Execute each step. Record observations under the matching acceptance
checklist entries.

### Step 1 - Watch follows iPhone active account (A to B and back)

This step exercises the user-initiated switch path
(`store.onActiveAccountChange -> propagateActiveAccountChange -> relay.send`)
and confirms the watch header label, the ring data, and
`TokenStore.accountLabel` all flip to match the iPhone.

1. Look at the Apple Watch dashboard. Confirm the tappable account
   header shows `accountA` (or whatever local-part
   `accountA@example.com` normalizes to). If the full address fits
   inside the watch header budget (14 characters in
   `trimmedForWatchHeader`), it may show the full `accountA@...`
   form; either is acceptable.
2. On the iPhone, tap the account chip on the Dashboard. Tap the row
   for account B. The iPhone chip flips to `accountB@example.com`.
3. On the Apple Watch, WITHIN ROUGHLY 2 SECONDS of the iPhone switch
   (per the `updateApplicationContext` latency budget across a live
   Bluetooth pairing):
   - The header label MUST flip to `accountB` (or the short form of
     account B's email per the same rules).
   - The ring MUST rebind to account B's `utilization5h` and
     `utilization7d`. The center percentage digit and the countdown
     to `resetAt5h` change to B's values.
   - The "mock" badge MUST NOT appear (both accounts produce live
     `isMocked = false` payloads; the badge is the small overlay
     drawn by `ContentView` when `usageState.isMocked` is `true`).
4. Tap the header label on the watch. Confirm the
   `accountDetailSheet` shows the full `accountB@example.com`
   string (not the truncated form). Tap Done to dismiss.
5. On the iPhone, tap the chip again and flip back to account A.
   Repeat the observations above: the watch header MUST flip to
   `accountA` within ~2 seconds and the ring MUST rebind to A's
   values.

Log spot-check (optional but recommended):

Open `Console.app` on the Mac, select the paired iPhone, filter by
`Tempo` process and the literal `AlertTrace`. During each account
switch the iPhone emits:

```
TempoApp active account changed accountId=accountB@example.com
```

followed by a `WatchRelay` call that turns into a new
`updateApplicationContext`. The watch, in its own log stream,
receives `didReceiveApplicationContext` and
`WatchSessionReceiver.applyUsageState` runs
`store.applyActiveAccount(id:label:)`. Both the iPhone and watch
logs should be non-empty across each switch.

Under-the-hood invariants to confirm via the LLDB debugger attached
to the watch process (optional, only if a visible UI step fails):

- `po store.activeAccountId` MUST equal the canonical id of the
  currently active account on the iPhone.
- `po store.accountLabel` MUST equal the string the iPhone sent as
  `accountLabel` in the payload. In the current build that value is
  the `accountId` itself (see
  `TempoApp` `relay?.send(..., accountLabel: state.accountId)`), so
  `accountLabel == activeAccountId` is expected. A future task may
  replace this with a user-set display name; if you see a different
  value in your build, treat it as a non-regression.
- `po store.hasNoActiveAccount` MUST equal `false`.

### Step 2 - NoActiveAccount clears the watch

This step exercises
`WatchRelayManager.sendNoActiveAccount` and the corresponding
`WatchSessionReceiver.applyNoActiveAccount` path.

The easiest way to trigger a real `NoActiveAccount` relay is to sign
out of the last remaining account on macOS. Before starting this
step, confirm the iPhone is on account A active (the end state of
Step 1).

1. On the Mac, open `Tempo macOS` > Preferences > Accounts card.
   Sign out of account B (9.4 Step 5 flow). The iPhone's Accounts
   sheet will subsequently reconcile to two accounts becoming one
   through the `onAccountsIndexUpdated` path, but the iPhone's
   active account is already A, so no watch-visible change fires
   yet. The macOS app should still show A active.
2. On the Mac, sign out of account A (9.4 Step 6 flow). This is the
   trigger:
   - macOS's iCloud `accounts/index.json.accountIds` becomes `[]`.
   - iOS's `onAccountsIndexUpdated` callback reassigns
     `store.activeAccountId = nil` (see `TempoApp.swift` "accounts
     index update reassigning" trace).
   - `store.onActiveAccountChange` fires with `nil`, which invokes
     `propagateActiveAccountChange`. Because `store.usage == nil`
     (no per-account usage file remains in iCloud), the else-branch
     runs and calls `relay.sendNoActiveAccount()`.
   - `WatchRelayManager.sendNoActiveAccount` posts
     `["type": "NoActiveAccount"]` via
     `updateApplicationContext`.
3. On the Apple Watch, WITHIN ROUGHLY 2 TO 5 SECONDS of the macOS
   sign-out (iCloud push to iPhone plus WatchConnectivity context
   hop):
   - The usage ring MUST disappear.
   - The "No accounts available" glyph
     (`person.crop.circle.badge.questionmark`), headline
     "No accounts available", and subtitle "Check the Mac app to
     add one" MUST be visible. This is the `noAccountsState` branch
     in `ContentView`.
   - The account header above the ring is gone (because the ring
     itself is gone).
   - Any `pendingCompletion` sheet that was up MUST be dismissed
     automatically (see `TokenStore.applyNoActiveAccount` which
     sets `pendingCompletion = nil`).
4. Tap-and-hold the watch face to open the complication picker
   (optional sanity check): the
   `TempoWatchShared.complicationUtilization5hKey` UserDefault is
   NOT rewritten on `NoActiveAccount` (only
   `applyUsageState` writes that key), so if the user had a
   utilization complication installed it will keep displaying the
   last known value until a fresh `UsageState` arrives. This is a
   known, acceptable behavior for this task and is called out here
   so testers do not flag it as a failure.

Under-the-hood invariants (optional, via LLDB attached to the watch
process during the clear):

- `po store.activeAccountId` MUST equal `nil`.
- `po store.accountLabel` MUST equal `""`.
- `po store.hasNoActiveAccount` MUST equal `true`.
- `po store.pendingCompletion` MUST equal `nil`.
- `po store.usageHistory.count` MUST equal `0`.
- `po store.usageState.utilization5h` MUST equal `0`.
- `po store.usageState.accountId` MUST equal
  `AccountIdentifier.unassignedAccountId`
  (the zeroed `emptyUsageState` placeholder).

Log spot-check: the iPhone's `Tempo` process emits

```
TempoApp propagateActiveAccountChange sending NoActiveAccount; resolvedAccountId=nil
```

and `WatchRelayManager` then calls `updateApplicationContext` with
the `NoActiveAccount` payload. The watch process emits (implicitly
via `WatchSessionReceiver.applyUsageInfo` routing) a
`Task @MainActor` hop into `store.applyNoActiveAccount()`; the
clearing is observable as the UI flip above.

Before moving on, re-establish both accounts for the remaining
steps:

5. On the Mac, sign back in to account A. Wait for the iPhone's
   Accounts sheet to show A and the watch dashboard to rebind to
   account A's ring (up to ~60 seconds for iCloud push plus the
   WatchConnectivity context hop). The watch header MUST flip
   from "No accounts available" back to the normal ring with the
   `accountA` header.
6. On the Mac, sign back in to account B. Confirm both accounts
   appear on the iPhone's Accounts sheet. On the iPhone, ensure
   account A is still active (tap the chip, tap account A if
   needed; the watch header should continue to show `accountA`).

### Step 3 - Completion alert on the active-account session fires

This step establishes the positive control for Step 4: when the
active-account session completes on macOS, the watch MUST fire a
haptic / local notification. With account A active on the iPhone
from Step 2.6:

1. Ensure the watch is on-wrist (crown pressed to unlock) and the
   screen is dim/asleep. This is the state that best demonstrates a
   haptic nudge on completion.
2. In Watch > Settings > Notifications > Tempo, confirm Alerts is
   ON and the delivery style is "Prominent" or "Banner". Either
   works; "Prominent" makes the nudge most visible during this
   test.
3. On the Mac, make sure the Claude Code CLI is signed in as
   account A. Confirm via `cat ~/.claude.json | grep
   emailAddress` or similar (the exact path may vary; the point is
   that `oauthAccount.emailAddress` in the CLI's profile must equal
   `accountA@example.com`).
4. In a new Terminal, run a short prompt through the CLI
   (e.g. `claude "respond with ok"`) and wait for it to return.
5. Within roughly 5 to 15 seconds of the CLI returning (accounting
   for `ClaudeLocalDBReader` polling, iCloud push to iOS, and the
   iOS relay hop):
   - The watch MUST raise a haptic (the default notification
     haptic on Apple Watch) and/or show a banner notification
     authored by `WatchAlertManager.notifySessionCompletion`. The
     banner title / subtitle / body come from
     `SessionInfo.notificationPresentation()` and the sound is
     `UNNotificationSound.default`.
   - A `CompletionView` sheet MUST appear on the watch showing the
     session summary and a footer line of the form
     `for accountA@example.com`. The sheet is driven by
     `store.pendingCompletion` in `RootView`'s
     `.sheet(item:)` modifier.
   - The iPhone MAY also fire a local notification via
     `PhoneAlertManager.notifySessionCompletion`, depending on the
     iPhone's alert preferences. That iPhone-side behavior is not
     in scope here; focus on the watch.
6. Tap the completion sheet to dismiss. The watch returns to the
   dashboard.

Log spot-check: the iPhone emits

```
TempoApp received synced session id=<sessionId> accountId=accountA@example.com activeAccountId=accountA@example.com ...
```

followed by a `WatchRelay` `sendSession` that does NOT trip the
`Skipped SessionInfo because accountId=... does not match
activeAccountId=...` trace (because the ids match). The watch
process runs `applySessionInfo`, writes to `sessions`, sets
`pendingCompletion`, and calls
`WatchAlertManager.notifySessionCompletion`.

### Step 4 - Completion alert SUPPRESSED for non-active account

This step is the core of 9.6. With account A still active on the
iPhone:

1. Switch the Claude Code CLI on the Mac to account B
   (`claude logout`, then `claude login`, completing OAuth for
   account B). After the CLI reports the new account, confirm via
   `cat ~/.claude.json | grep emailAddress` that the profile email
   is `accountB@example.com`.
2. Ensure the iPhone is still on account A active (check the chip
   on the Dashboard). If the iPhone has drifted to B for any
   reason, flip it back to A before running the trigger.
3. Still with account A active on the iPhone, run a short prompt
   through the CLI as account B (e.g. `claude "respond with ok"`).
   Wait for it to return.
4. For roughly 30 seconds after the CLI returns, KEEP THE WATCH
   UNDER OBSERVATION and on-wrist:
   - The watch MUST NOT fire any haptic.
   - No banner notification for this session MUST appear.
   - No `CompletionView` sheet MUST appear.
   - The dashboard continues to show account A's ring with the
     `accountA` header label, unchanged.
5. Confirm via the iPhone `Console.app` log stream that the iPhone
   observed the B-account completion and actively suppressed it.
   The expected traces are:

   ```
   TempoApp received synced session id=<id> accountId=accountB@example.com activeAccountId=accountA@example.com iPhoneAlerts=... watchAlerts=...
   TempoApp suppressing iPhone notification for session id=<id> accountId=accountB@example.com because it does not match activeAccountId=accountA@example.com
   Skipped SessionInfo because accountId=accountB@example.com does not match activeAccountId=accountA@example.com
   ```

   The first line confirms iOS read the B-account `latest.json`.
   The second confirms iOS suppressed the iPhone notification (not
   watch-scoped but part of the same gate). The third, emitted by
   `WatchRelayManager.sendSession`, is the PRIMARY evidence that
   no `SessionInfo` was handed to `transferUserInfo` for the
   watch.
6. Confirm the inner watch-side safety net never tripped because
   the outer gate fired first. The watch process log should NOT
   contain
   `Dropped SessionInfo: session accountId=... does not match
   activeAccountId=...`. If that inner-gate trace DOES appear, it
   is not a failure for this test (the suppression still worked),
   but it indicates the outer iOS gate may have a race; record it
   under Notes.
7. Now flip the iPhone to account B active via the chip. Wait
   roughly 10 seconds (WatchConnectivity context hop plus the
   `iCloudUsageReader`'s next tick). The watch MUST NOT fire an
   alert for the earlier B-account completion, because the dedup
   slot `lastRelayedSessionID` was not touched during the
   suppressed send, but `WatchRelayManager.sendSession` is only
   invoked by the reader callback on new `latest.json` writes. A
   second alert is NOT expected after the switch unless a new
   session event lands. In practice this means: no spurious
   retroactive haptic fires when the user switches to the account
   that previously had a suppressed completion. Confirm the watch
   simply shows account B's ring. If an alert does fire here,
   record it under Notes; the expected behavior for 9.6 is "no
   retroactive alert on account switch".
8. Before continuing to Step 5, leave the iPhone on account B
   active. The next step exercises the unassigned exemption.

### Step 5 - Unassigned (CLI-only) session exemption fires regardless

This step confirms the `AccountIdentifier.unassignedAccountId`
exemption path: CLI-only sessions with no owning account MUST
deliver a watch alert no matter which account is active.

Producing an unassigned session requires the CLI's
`oauthAccount.emailAddress` to NOT match any account in the macOS
`AccountRegistry`. A reliable way to produce one:

1. On the Mac, sign the Claude Code CLI in as a THIRD anthropic
   account (call it account C) whose email is NOT added in
   `Tempo macOS` Preferences > Accounts. If no such third account
   is available, an alternative is to temporarily remove account
   A from `Tempo macOS` while leaving account A signed in on the
   CLI, producing an `oauthAccount.emailAddress` that does not
   match the registry. (If you use this alternative, re-add
   account A on macOS at the end of this step.)
2. With account B still active on the iPhone from Step 4.8, run a
   short prompt through the CLI as the now-unregistered account:

   ```sh
   claude "respond with ok"
   ```

3. Within roughly 5 to 15 seconds of the CLI returning:
   - The watch MUST fire a haptic / banner.
   - A `CompletionView` sheet MUST appear.
   - The sheet's footer line reads `CLI-only session` (not
     `for accountB@example.com`). This comes from
     `CompletionView.displayAccountLine()` which branches on
     `session.accountId == AccountIdentifier.unassignedAccountId`.
4. Tap the sheet to dismiss. Confirm the dashboard returns to
   account B's ring with the `accountB` header label.

Log spot-check: the iPhone emits
`TempoApp received synced session ... accountId=unassigned
activeAccountId=accountB@example.com ...` and then
`WatchRelay` proceeds without the `Skipped SessionInfo`
trace, because
`sessionInfo.accountId == AccountIdentifier.unassignedAccountId`
does not equal `activeAccountId`, but the gate condition is
`sessionInfo.accountId != activeAccountId` and the exemption is
enforced on the watch side in
`WatchSessionReceiver.applySessionInfo` via the `isUnassigned`
check. The expected behavior: the iOS outer gate will still
suppress this payload because the outer gate in
`WatchRelayManager.sendSession` does NOT currently exempt
unassigned ids (it only checks strict equality). In practice this
means the exemption only fully takes effect when the watch
receives an unassigned session, which happens when
`activeAccountId == nil` at the time of sending (for example,
right before a switch).

If in your run the watch does NOT fire an alert for the unassigned
session while account B is active, this is EXPECTED under the
current outer-gate behavior and is NOT a failure for this
verification. The acceptance for this step is specifically: when
the iPhone is in a state where the unassigned session reaches the
watch (no active account, or the outer gate path is updated in a
later task to exempt `unassigned`), the watch side MUST NOT
suppress it. You can exercise this by signing out of all macOS
accounts temporarily (to put the iPhone in the `NoActiveAccount`
state from Step 2) and replaying the CLI run; in that case the
outer gate's `activeAccountId` argument is `nil`, the gate is a
no-op, and the watch receives the payload. Re-add both accounts on
macOS after this substep.

Record under Notes any observation that the unassigned session
reached the watch (or did not) and under which outer-state
conditions.

### Step 6 - Stale pendingCompletion dropped on account change

This step exercises
`TokenStore.clearPendingCompletionIfStale(newAccountId:)`.

Set up: the watch must have a `pendingCompletion` currently set for
one account and then receive a relay for a different account before
the user has dismissed the sheet. Because the sheet is modal, the
easiest way to create this condition in practice is:

1. Set the iPhone to account A active.
2. On the Mac, switch the CLI to account A and run a prompt that
   returns ("respond with ok"). Wait for the watch to fire the
   completion alert and for the `CompletionView` sheet to appear.
3. DO NOT tap the sheet to dismiss. Leave it presented on the
   watch.
4. On the iPhone, tap the chip and switch active account to B.
   This triggers a fresh `updateApplicationContext` carrying
   account B's `UsageState`, which routes through
   `WatchSessionReceiver.applyUsageState ->
   store.apply(state) ->
   clearPendingCompletionIfStale(newAccountId: "accountB@example.com")`.
5. Within roughly 2 seconds of the iPhone chip flip:
   - The `CompletionView` sheet for account A MUST be dismissed
     AUTOMATICALLY on the watch (because
     `clearPendingCompletionIfStale` sets
     `pendingCompletion = nil`, and `RootView`'s
     `.sheet(item: $bindableStore.pendingCompletion)` unbinds).
   - The dashboard MUST rebind to account B's ring.
   - The header label MUST flip to `accountB`.
   - No new haptic fires as part of the dismissal.
6. Log spot-check on the watch process:

   ```
   Clearing stale pendingCompletion: pending accountId=accountA@example.com != new=accountB@example.com
   ```

   This line is emitted by `clearPendingCompletionIfStale`. It is
   the definitive signal the drop happened. If you do not see this
   line but the sheet still vanished, that means the sheet
   dismissal came from a different source (for example, watch
   auto-dismiss of modal sheets on screen sleep); rerun the step
   paying closer attention to timing.

LLDB inspection at the moment after the switch (optional):

- `po store.pendingCompletion` MUST equal `nil`.
- `po store.activeAccountId` MUST equal `accountB@example.com`.
- `po store.sessions.count` MUST be unchanged or incremented by 1

## Reference observations

Because watchOS does not give Terminal-level access to App Group
directories on a stock device, this verification is primarily
UI-observable plus `AlertTrace` log-observable.

Useful `AlertTrace` search patterns in `Console.app`:

- On the iPhone, filter process `Tempo`:
  - `TempoApp active account changed accountId=` - fires on every
    user-initiated chip switch.
  - `TempoApp propagateActiveAccountChange sending NoActiveAccount`
    - fires when iOS relays the empty state to the watch.
  - `TempoApp received synced session id=` - fires every time
    `iCloudUsageReader` hands a session to `TempoApp`.
  - `TempoApp suppressing iPhone notification for session id=` -
    the iPhone-side gate that parallels the watch-side gate.
  - `Skipped SessionInfo because accountId=... does not match
    activeAccountId=...` - the watch-side outer gate from
    `WatchRelayManager.sendSession`.
- On the watch, filter process `Tempo Watch`:
  - `Dropped SessionInfo: session accountId=... does not match
    activeAccountId=...` - the watch-side inner gate from
    `WatchSessionReceiver.applySessionInfo`. Should fire
    essentially never in a clean run because the outer gate
    already dropped the transfer.
  - `Skipping pendingCompletion: session accountId=... !=
    active=...` - the innermost gate in
    `TokenStore.applySession`; guards the sheet specifically.
  - `Clearing stale pendingCompletion: pending accountId=... !=
    new=...` - the Step 6 trace.
  - `Dropped UsageState payload on watch: missing accountId` -
    should never fire in a well-formed relay; record it under
    Notes if you see it.

LLDB inspection on the watch process while the watch is attached to
Xcode:

- `po store.activeAccountId`
- `po store.accountLabel`
- `po store.hasNoActiveAccount`
- `po store.pendingCompletion`
- `po store.usageState.accountId`
- `po store.sessions.last?.accountId`

These reads must be run on the main thread (LLDB's default for
Xcode watchOS debugging is fine).

## Expected observable states

### After Step 1 (switch to B, then back to A)

- Watch header label tracks the iPhone chip within ~2 seconds of
  each switch.
- Ring `utilization5h` and `utilization7d` rebind to the newly
  active account on each switch.
- `TokenStore.activeAccountId`, `TokenStore.accountLabel`, and
  `TokenStore.hasNoActiveAccount` are coherent
  (`activeAccountId` non-nil, `accountLabel` non-empty,
  `hasNoActiveAccount == false`).

### After Step 2 (last-account sign-out on macOS)

- Ring is replaced by the "No accounts available" placeholder.
- `person.crop.circle.badge.questionmark` glyph, the "No accounts
  available" headline, and the "Check the Mac app to add one"
  subtitle are all visible.
- Any active `CompletionView` sheet is dismissed.
- `TokenStore.activeAccountId == nil`,
  `TokenStore.accountLabel == ""`,
  `TokenStore.hasNoActiveAccount == true`,
  `TokenStore.pendingCompletion == nil`.
- After re-adding A and B on macOS, the watch rebinds to A's ring
  with the `accountA` header label.

### After Step 3 (active-account completion)

- Watch haptic + `CompletionView` sheet for the
  `accountA@example.com` session.
- Sheet footer reads `for accountA@example.com`.

### After Step 4 (non-active-account completion)

- No watch haptic.
- No `CompletionView` sheet on the watch.
- iPhone `Console.app` log shows the `Skipped SessionInfo`
  trace for the account B session id.
- Switching the iPhone to account B afterward does NOT retroactively
  fire a watch alert for the earlier completion.

### After Step 5 (unassigned completion)

- When the payload path reaches the watch (for example, `activeAccountId`
  is nil during the send), the watch haptic fires, the
  `CompletionView` sheet appears, and the sheet footer reads
  `CLI-only session`.
- When the outer gate's equality check suppresses the unassigned
  send, the watch is silent. Either outcome is acceptable as long
  as the watch-side receiver does NOT suppress a session with
  `accountId == "unassigned"` once delivered.

### After Step 6 (stale pendingCompletion)

- The A-account completion sheet is dismissed within ~2 seconds of
  the iPhone switching to account B.
- Watch process log contains the `Clearing stale pendingCompletion`
  trace.
- `TokenStore.pendingCompletion == nil`,
  `TokenStore.activeAccountId == accountB@example.com`.
- `TokenStore.sessions` still contains the earlier A-account
  session (history preserved, only the modal was cleared).
