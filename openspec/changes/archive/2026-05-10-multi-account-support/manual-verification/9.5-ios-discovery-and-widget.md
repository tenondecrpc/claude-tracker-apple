# 9.5 Manual Verification - iOS Discovery, Account Chip Switching, Activity Filtering, Widget Pinning

This document is the manual verification procedure for task 9.5 of the
`multi-account-support` change. It covers the iOS companion app
(`Tempo` scheme) running on a real iPhone and the iOS widget extension
(`Tempo iOS Widget` target). A human tester MUST execute each step in
order and mark the acceptance checklist at the end.

Objective: confirm that the iOS companion picks up both accounts via
iCloud, that the dashboard account chip flips the active account and
rebinds the dashboard + Activity views, that the `ios.activeAccountId`
UserDefault survives an app relaunch, that the iOS home screen widget
can be pinned to a specific account via `SelectAccountIntent`, that it
falls back to the active account with an "Account removed" badge when
the pinned account vanishes on macOS, and that tapping a widget pinned
to account A launches the app on account A via the
`TempoWidgetRoute` deep link.

This task is iOS-only; it consumes the per-account iCloud tree produced
by macOS in 9.4 and does not itself change Keychain or iCloud writes.
Do NOT attempt to exercise this on the iOS Simulator: `iCloudUsageReader`
early-exits on `targetEnvironment(simulator)` because the simulator
cannot access the ubiquity container, and widgets run on a real device
widget extension. Use a physical iPhone.

## Preconditions

Environment:

- A physical iPhone running iOS 26 or later (matching the app's
  deployment target). The iPhone MUST be signed into the same iCloud
  account as the macOS test machine from 9.4 and MUST have iCloud Drive
  enabled in Settings > Apple Account > iCloud > iCloud Drive.
- The `Tempo` scheme built fresh from the current branch in Xcode and
  installed on the iPhone via `Cmd+R` with the device selected. Do NOT
  install a TestFlight build for this verification.
- The `Tempo iOS Widget` extension is included in the install (it ships
  as an embedded extension of `Tempo`, so a clean install of `Tempo`
  brings it along; verify by adding a widget in Step 5 below).
- 9.4 has been completed successfully on the paired Mac. Specifically,
  the iCloud tree under
  `~/Library/Mobile Documents/iCloud~com~tenondev~tempo~claude/Documents/Tempo/accounts/`
  must currently contain two real accounts (A and B) with
  `account.json`, `usage.json`, and `usage-history.json` present for
  both. `accounts/index.json` must list both accountIds in registry
  order.
- The macOS `Tempo` build is running (menu bar icon visible) throughout
  this run so polls keep landing on iCloud for both accounts and the
  tester can use it to remove account A during Step 6.
- Two real Anthropic accounts already signed in on the Mac with
  canonical ids:
  - Account A: `accountA@example.com` (substitute real email).
  - Account B: `accountB@example.com` (substitute real email).
- The iPhone has a usable cellular or Wi-Fi connection so iCloud can
  push the accounts tree to the device.
- Access to the iPhone console via
  `Console.app` (Mac, with the iPhone selected) or `idevicesyslog` if
  the tester wants to watch the `AlertTrace` DevLog stream. Not
  strictly required; all visible UI states are checkable by eye.

Starting state:

- Delete any previous install of `Tempo` from the iPhone
  (long-press icon > Remove App > Delete App). This guarantees
  `UserDefaults` under the iOS app's bundle id is empty and that
  `ios.activeAccountId` is unset at launch so we can observe the
  first-discovered-account default behavior.
- Remove any existing Tempo widget instances from the home screen to
  keep the widget test in Step 5 uncluttered.
- Plug the iPhone into the Mac and unlock it so Xcode can install the
  fresh build. Trust the Mac on the iPhone if prompted.

## Definitions

- iCloud container identifier consumed by iOS:
  `iCloud.com.tenondev.tempo.claude`. On the iPhone, iCloud Drive files
  for the container are visible under Files > Browse > iCloud Drive >
  Tempo for Claude but are not directly inspectable via Terminal. All
  iOS-side verification uses the in-app UI and widget state.
- App Group used by the iOS widget extension and host app:
  `group.com.tenondev.tempo.claude.ioswidget`. Snapshot store layout
  under that App Group:
  - `Library/Application Support/Tempo/accounts/<percentEncodedAccountId>/tempo.widget.snapshot.json`
  - `Library/Application Support/Tempo/active-account.json` (pointer
    file; written by `TempoWidgetSnapshotStore`)
- iOS UserDefaults keys used in this flow:
  - `ios.activeAccountId` (String; canonical accountId or missing).
    Written by `IOSAppStore.activeAccountId.didSet` when the user taps
    a row in the Accounts sheet. Missing key means "resolve via
    `knownAccountIds.first`".
- Deep link scheme: `tempoforclaude`. Widget tap URL produced by
  `TempoWidgetRoute`:
  `tempoforclaude://dashboard?accountId=<percentEncodedAccountId>`.
  When a widget is pinned to a specific account via
  `SelectAccountIntent`, the URL embeds that account's id. When the
  pin is absent the URL uses the snapshot's own accountId (which
  equals the active account at render time).
- Account chip: the capsule button at the top of the Dashboard tab
  that shows the active account's canonical id and a chevron. Tapping
  it opens the Accounts sheet (`AccountsSheetView`).
- Activity tab: the second tab (`ActivityTabView`). It reads
  `store.filteredHistorySnapshots`, which delegates to
  `iCloudReader.historyByAccount[store.resolvedAccountId]`. There is
  no other history source, so switching the active account MUST
  refilter this tab automatically.
- Widget "Account removed" badge: small warning capsule rendered in
  the top-right of the widget chrome when a `SelectAccountIntent`
  configuration points at an accountId that no longer has a snapshot
  on disk. In that case the provider falls back to the active
  account's snapshot (task 8.3).

## Procedure

Execute each step. Record observations under the matching acceptance
checklist entries.

### Step 1 - Fresh install and first launch

1. Install `Tempo` from Xcode to the iPhone via `Cmd+R`. Wait for the
   BootView splash to dismiss (a circular progress spinner over the
   launch logo).
2. Confirm the app lands on the Dashboard tab with the account chip
   at the top of the view.
3. Because this iPhone has no persisted `ios.activeAccountId`, the
   chip MUST display the first accountId in
   `iCloudReader.knownAccountIds`, which mirrors the macOS registry
   order. With 9.4's setup that is account A
   (`accountA@example.com`).
4. Confirm that within roughly 30 to 90 seconds (depending on iCloud
   push latency) the dashboard ring populates with account A's
   `utilization5h` / `utilization7d`, the Sync Status card flips from
   "Waiting for iCloud file" to "Updated ...", and the "Connect via
   Mac App" waiting card disappears.
5. Tap the Activity tab. Confirm the history chart renders account
   A's data (the segmented "Range" control defaults to `last7Days`
   per `UsageHistoryRange`). If the history is empty, scroll back to
   Dashboard, wait another 30 seconds for iCloud to push
   `usage-history.json`, and retry.

Notes on discovery:

- `iCloudUsageReader.start()` dispatches container resolution off the
  main thread and then installs an `NSMetadataQuery` with predicate
  `NSMetadataItemFSNameKey IN { "usage.json", "usage-history.json",
  "latest.json", "account.json", "index.json", ... }` scoped to
  `NSMetadataQueryUbiquitousDocumentsScope`. Both account directories
  under `Tempo/accounts/` MUST be matched by that predicate, which
  populates `iCloudReader.usageByAccount`,
  `iCloudReader.historyByAccount`, and `iCloudReader.knownAccountIds`.
- Discovery also reads `Tempo/accounts/index.json` and fires
  `onAccountsIndexUpdated`. The coordinator in `TempoApp.swift`
  reconciles this against `store.activeAccountId` and, when the id is
  missing or stale, falls back to `iCloudReader.knownAccountIds.first`.

### Step 2 - Open Accounts sheet and confirm both accounts discovered

1. On the Dashboard tab, tap the account chip at the top. The
   `AccountsSheetView` sheet presents in a NavigationStack titled
   "Accounts".
2. Confirm the sheet lists two rows, one per account, in registry
   order (A then B, matching the order in
   `accounts/index.json.accountIds`).
3. Each row displays:
   - The canonical accountId as the primary label
     (displayName materialization from `account.json` is a future
     enhancement; today the row label is the accountId itself).
   - A subtitle of the form "Updated X ago" if the reader has
     observed a `usage.json` for that account, or "No data yet"
     otherwise.
   - A trailing checkmark on the currently active row.
4. Confirm the row for account A carries the checkmark (it was the
   first-discovered default in Step 1) and the row for account B
   does NOT.
5. Confirm the footer section reads "To add an account, open Tempo
   for Claude on your Mac and sign in." iOS does NOT expose an Add
   affordance; adding accounts is macOS-only.
6. Tap Done to dismiss the sheet without changing the active account.

### Step 3 - Switch active account via chip, confirm rebind and persistence

1. Tap the account chip again to reopen the sheet.
2. Tap the row for account B. The sheet dismisses immediately.
3. Confirm the chip label flips to account B.
4. Confirm the dashboard ring, metric pills, and Burn Rate card
   rebind to account B's usage. This happens because
   `IOSAppStore.setActiveAccount(accountId:)` writes
   `activeAccountId` (which updates `resolvedAccountId`, which
   drives `store.usage` to return account B's `UsageState` from
   `iCloudReader.usageByAccount`).
5. Switch to the Activity tab. Confirm the chart and summary pills
   (Avg Session, Avg Weekly, Points) reflect account B's history.
   Crucially, the chart MUST NOT merge A's and B's history; the
   view reads only `store.filteredHistorySnapshots` which is
   filtered by `resolvedAccountId`.
6. Background the app (swipe up) and fully terminate it (swipe up
   on its card in the app switcher).
7. Relaunch `Tempo` from the home screen. Wait for BootView to
   complete.
8. Confirm the account chip shows account B on first render of the
   Dashboard. This proves `ios.activeAccountId` was persisted to
   UserDefaults and restored by `IOSAppStore.init`.
9. Optional inspection on the Mac (the iPhone's UserDefaults is not
   directly inspectable from the iPhone without a debugger). Attach
   the Xcode debugger to a later run and evaluate
   `po UserDefaults.standard.string(forKey: "ios.activeAccountId")`
   in the LLDB console; the value MUST equal the canonical accountId
   for account B.
10. Switch back to account A using the chip (tap chip, tap account A
    row). Confirm the chip, dashboard ring, and Activity tab all
    rebind to account A. Leave account A as the active selection
    before moving on.

### Step 4 - Confirm Activity tab is strictly filtered, not merged

1. On Dashboard, note the current 5H percent for account A from the
   metric pill.
2. On the Mac, open the Stats detail window and confirm account A's
   5H percent there matches (or is within a minute of) the iPhone
   value.
3. On the iPhone, switch to Activity.
4. In the Range segmented control, select `Last 24 Hours`. Wait for
   the chart to re-render.
5. Confirm the chart plots only the A-keyed history series. The
   Points stat pill in the Summary card MUST equal the number of
   snapshots in `iCloudReader.historyByAccount["accountA@example.com"]`
   that fall inside the last 24 hours. It MUST NOT equal the sum of
   A's and B's points.
6. Tap the chip and switch to account B. Confirm the chart redraws
   with B's data only. Toggle Range back to `Last 7 Days` and
   confirm the x-axis extent changes but the underlying series
   still belongs to B only.
7. Switch back to account A before Step 5.

### Step 5 - Widget with pinned account A and pinned account B

1. Long-press an empty spot on the iPhone home screen until icons
   jiggle. Tap the `+` (Add Widget) control.
2. Search for "Tempo". The widget family should expose three
   configurations: "Tempo Usage" (systemSmall ring), "Tempo Summary"
   (systemMedium), and "Tempo Compact" (systemSmall).
3. Add the "Tempo Usage" ring widget (systemSmall) to the home
   screen. The freshly added widget SHOULD render in a "follow
   active account" state: the provider reads
   `TempoWidgetSnapshotStore.read(platform: .iOS)` because the
   intent's `account` parameter is unset, and that call resolves
   via the `active-account.json` pointer to account A's snapshot.
4. Long-press the widget > Edit Widget. The configuration sheet
   should show a single field: "Account".
5. Tap the Account field. The picker lists the accountIds returned
   by `AccountEntityQuery.suggestedEntities()`, which in turn reads
   `TempoWidgetSnapshotStore.knownAccountIds(platform: .iOS)`.
   Confirm both `accountA@example.com` and `accountB@example.com`
   appear in the list.
6. Select account A. Tap outside the configuration sheet to commit.
7. Confirm the widget renders account A's ring. The tap URL it
   carries (set by `TempoIOSWidgetChrome` via `widgetURL(...)`)
   embeds `accountId=accountA@example.com` in the deep link query.
8. Repeat steps 1 to 6 to add a second "Tempo Usage" widget (or the
   Summary variant for variety), and this time pin it to account B
   via the picker.
9. Confirm both widgets render their respective accounts side by
   side. The A widget's ring reflects account A's usage; the B
   widget's ring reflects account B's usage. They MUST NOT both
   show the same account.

Notes on snapshot store behavior:

- `IOSAppStore` + `AppCoordinator` in `TempoApp.swift` call
  `TempoWidgetSnapshotStore.write(_:platform: .iOS)` for every
  account whose `UsageState` arrives via iCloud, and they update the
  `active-account.json` pointer only when the active account
  changes. So as long as iCloud has pushed both `usage.json` files
  to the iPhone (Step 1 and Step 2 validated this), both per-account
  snapshots are present in the App Group container and both pins
  resolve correctly.

### Step 6 - Pinned-account fallback when account A is removed on macOS

1. On the Mac, sign out of account A via Preferences > Accounts >
   Sign out (per 9.4's Step 5 procedure). Confirm the macOS
   `AccountRemovalService` removes the iCloud
   `Tempo/accounts/accountA@example.com/` directory and updates
   `accounts/index.json` to contain only account B.
2. On the iPhone, wait up to roughly 60 seconds for iCloud to push
   the index and directory removal to the device. Watch the A
   widget on the home screen.
3. Expected: the A widget renders a small "Account removed" badge
   in its top-right corner and its body shows account B's usage
   (the active account). This is the fallback path in
   `TempoIOSProvider.resolveSnapshot(for:)`: the pinned
   `configuration.account?.id` equals `accountA@example.com`, but
   `TempoWidgetSnapshotStore.read(accountId:platform:)` returns
   `nil` because the iOS coordinator reconciled the accounts index
   update and no longer has an account A snapshot, so the provider
   returns the active-account snapshot plus
   `configuredAccountRemoved: true`.
4. The B widget MUST continue to render account B's usage with no
   badge (it was pinned to B, which is still present).
5. Inside the app, the account chip MUST have flipped to account B
   automatically. This is `iCloudReader.onAccountsIndexUpdated` ->
   `IOSAppStore.setActiveAccount(accountId:)` reassigning when the
   persisted id disappears from the known set.
6. The Accounts sheet MUST now show only account B, with the
   footer "To add an account, open Tempo for Claude on your Mac and
   sign in." visible.

If the iOS coordinator has not yet picked up the removal after 60
seconds, force it by: background the app, pull the iPhone off
Wi-Fi for 5 seconds, restore Wi-Fi, then foreground the app. The
`onBecomeActive` hook calls `iCloudReader.restart()` which rebuilds
the metadata query and re-reads the index.

### Step 7 - Deep link: tap widget pinned to A, expect app on A

Before continuing, re-add account A on the Mac so Step 7 tests the
deep link with a valid pin. On the Mac, Welcome window > Sign In
with account A's credentials. Wait until the iPhone's Accounts
sheet shows both A and B again (up to 60 seconds).

Re-pin the A widget to account A via long-press > Edit Widget >
Account > A (the widget will still display the old pin id, but the
entity query now resolves it again, so the "Account removed" badge
will clear).

1. With the A widget now rendering account A and no badge, make
   sure the iOS app is fully backgrounded (swipe up on its card
   in the app switcher).
2. Actually terminate the app so the launch is cold: double-swipe
   on the Tempo card in the app switcher.
3. On the iPhone, change the iOS active account to account B by
   opening the app briefly, tapping the chip, selecting B,
   and closing the app again. This sets
   `ios.activeAccountId = accountB@example.com`.
4. Terminate the app once more so the next launch is cold.
5. Tap the A-pinned widget.
6. Expected: the app cold-launches and lands on the Dashboard
   showing account A. This happens because `TempoApp.onOpenURL`
   parses `tempoforclaude://dashboard?accountId=accountA@example.com`
   via `TempoWidgetRoute(url:)` and, seeing that
   `route.accountId` does not equal `store.resolvedAccountId`,
   calls `store.setActiveAccount(accountId: accountA@example.com)`
   before the ContentView renders.
7. Confirm the account chip reads `accountA@example.com` on first
   draw. Confirm the Activity tab also renders account A's
   history.
8. Optional: tap the B-pinned widget to verify the same deep link
   flow flips the active account back to account B. The same URL
   parsing logic runs with `accountId=accountB@example.com` and
   the chip, dashboard, and Activity all rebind to B.

## Reference observations

Because iOS does not give Terminal-level access to the ubiquity
container or App Group directories on a stock device, this
verification is primarily UI-observable. The following signals are
useful when a step appears to fail:

- `Console.app` filtered to the `Tempo` process on the iPhone. The
  app emits `DevLog.trace("AlertTrace", ...)` lines with messages
  like:
  - `"iCloudUsageReader saw metadata item name=usage.json path=..."`
  - `"iCloudUsageReader decoded accounts index path=... count=2"`
  - `"TempoApp active account changed accountId=accountB@example.com"`
  - `"TempoApp accounts index update reassigning activeAccountId from ... to ..."`
  These confirm per-account iCloud reads and active-account changes
  at the protocol level.
- Xcode LLDB attached to a running instance can inspect:
  - `po UserDefaults.standard.string(forKey: "ios.activeAccountId")`
  - `po iCloudReader.knownAccountIds`
  - `po iCloudReader.usageByAccount.keys.sorted()`
  - `po TempoWidgetSnapshotStore.knownAccountIds(platform: .iOS)`
  - `po TempoWidgetSnapshotStore.readActiveAccountId(platform: .iOS)`
  Use these only if a visible UI step fails and the root cause is
  not obvious. None of them are required to mark the checklist.
- Widget freshness: the widget chrome renders a small clock icon
  plus a relative time label. When iCloud push is delayed, the
  widget may show stale data for a minute or two. Do not treat a
  brief staleness as a failure; wait 60 seconds and retry the
  step.

## Expected observable states

### After Step 1 (fresh install, first launch)

- Chip label: `accountA@example.com`.
- Dashboard ring: populated within ~60s (both utilizations > 0
  assuming A has any usage; a brand-new account shows 0% which is
  also valid).
- Sync Status row `Usage`: "Updated ..." with a recent timestamp.
- Sync Status row `History`: "Updated ..." with a recent
  timestamp (may lag by a few seconds).

### After Step 2 (Accounts sheet)

- Sheet shows two rows (A, B) in that order.
- Row A has a trailing checkmark; Row B does not.
- Footer: "To add an account, open Tempo for Claude on your Mac
  and sign in."

### After Step 3 (switch to B, relaunch)

- Chip label: `accountB@example.com`.
- `UserDefaults["ios.activeAccountId"]` equals
  `accountB@example.com`.
- Dashboard rebinds to B's usage; Activity rebinds to B's history.
- After relaunch, chip still shows B.

### After Step 4 (Activity filtering)

- Points stat in Activity summary card for the current range
  reflects only the active account's history count; NOT the sum
  of A and B.

### After Step 5 (two widgets, different pins)

- Two ring widgets on the home screen: one labeled/reading A, one
  labeled/reading B.
- Neither widget shows the "Account removed" badge.
- Widget tap URLs embed `accountId=<pinned accountId>` (verified
  implicitly by Step 7 deep link behavior).

### After Step 6 (account A removed on macOS)

- A widget: renders B's usage with a visible "Account removed"
  badge in the top-right.
- B widget: unchanged, renders B's usage, no badge.
- In-app chip: `accountB@example.com`.
- Accounts sheet: single row for B with the checkmark; footer
  unchanged.

### After Step 7 (re-add A, tap A widget)

- Cold launch from A widget results in the app showing account A
  on Dashboard and Activity, regardless of the pre-tap
  `ios.activeAccountId` value.
- `UserDefaults["ios.activeAccountId"]` equals
  `accountA@example.com` after the launch completes.

## Acceptance checklist

Check every box before marking task 9.5 complete. If any box cannot be
checked, record the observation under `Notes` and leave the task in
the in-progress state.

Step 1 (discovery and default active account):

- [ ] Fresh install of `Tempo` on a physical iPhone launches without
      crashing and presents the Dashboard.
- [ ] Account chip displays account A on first render.
- [ ] Dashboard ring populates with account A's usage within roughly
      60 seconds of launch.
- [ ] Sync Status rows report "Updated ..." (not "Waiting for iCloud
      file" or "Stale since ...") once sync completes.
- [ ] No "Connect via Mac App" waiting card remains visible after
      sync completes.

Step 2 (Accounts sheet lists both accounts):

- [ ] Tapping the chip opens an Accounts sheet titled "Accounts".
- [ ] The sheet lists exactly two account rows: A and B, in that
      order.
- [ ] Row A shows a trailing checkmark; Row B does not.
- [ ] Each row shows an "Updated X ago" subtitle after iCloud push.
- [ ] Footer reads "To add an account, open Tempo for Claude on
      your Mac and sign in."

Step 3 (switch active account via chip, persistence):

- [ ] Tapping the B row in the sheet dismisses the sheet and flips
      the chip label to account B.
- [ ] Dashboard ring, metric pills, and Burn Rate card rebind to
      account B's `UsageState`.
- [ ] Activity tab chart and summary rebind to account B's
      history.
- [ ] After force-quit and relaunch, the chip still shows account B.
- [ ] (Optional) LLDB inspection confirms
      `UserDefaults.standard.string(forKey: "ios.activeAccountId")`
      equals `accountB@example.com`.
- [ ] Switching back to account A in the sheet restores A as active.

Step 4 (Activity strictly filtered, not merged):

- [ ] Changing the Range control to `Last 24 Hours` on Activity
      renders a chart whose Points count matches only the active
      account's per-account history slice.
- [ ] Switching accounts via the chip while on Activity redraws
      the chart for the other account alone (no merged series).
- [ ] Switching Range to `Last 7 Days` preserves the single-account
      filter.

Step 5 (widget pinning via SelectAccountIntent):

- [ ] Three Tempo widget configurations are available from the
      home-screen Add Widget flow: Tempo Usage (small ring), Tempo
      Summary (medium), Tempo Compact (small).
- [ ] The widget configuration sheet exposes an Account picker.
- [ ] The Account picker lists both `accountA@example.com` and
      `accountB@example.com`.
- [ ] Selecting account A on widget instance 1 and account B on
      widget instance 2 results in two distinct widget faces on
      the home screen, each rendering its pinned account's usage.
- [ ] Neither widget shows the "Account removed" badge at this
      point.

Step 6 (pinned-account-vanished fallback):

- [ ] Signing out of account A on macOS causes the A-pinned widget
      to switch to displaying account B's usage within roughly 60
      seconds of the macOS sign-out.
- [ ] The A-pinned widget displays a visible "Account removed"
      badge in its top-right corner while falling back.
- [ ] The B-pinned widget continues to display account B's usage
      and does NOT show the "Account removed" badge.
- [ ] The in-app account chip automatically flips to
      `accountB@example.com` without any user interaction.
- [ ] The Accounts sheet now lists only account B.

Step 7 (deep link: widget tap activates pinned account):

- [ ] After re-adding account A on macOS, the "Account removed"
      badge clears on the A-pinned widget.
- [ ] With `ios.activeAccountId = accountB@example.com` persisted
      and the app fully terminated, tapping the A-pinned widget
      cold-launches `Tempo` onto account A.
- [ ] The account chip reads `accountA@example.com` on the first
      Dashboard draw after the deep-link launch.
- [ ] The Activity tab renders account A's history after the
      deep-link launch.
- [ ] Tapping the B-pinned widget after a subsequent termination
      flips the app back to account B via the same deep link
      mechanism.

Global invariants (check once at the end):

- [ ] At no point during the run did the iOS app merge account A's
      and account B's usage in any single dashboard, Activity
      chart, or widget face.
- [ ] At no point did the iOS app read from the legacy flat paths
      `Tempo/usage.json`, `Tempo/usage-history.json`, or
      `Tempo/latest.json`. (Inspect via `Console.app` filtering
      `AlertTrace`: all decoded paths must live under
      `Tempo/accounts/<id>/`.)
- [ ] `ios.activeAccountId` in iOS UserDefaults is either a
      canonical accountId that is currently in
      `iCloudReader.knownAccountIds` or is unset. It is never
      a stale id after the coordinator reconciles with an updated
      `accounts/index.json`.

  (one screenshot per account).
- The Accounts sheet.
- The Activity tab with a non-empty chart and the Points summary
